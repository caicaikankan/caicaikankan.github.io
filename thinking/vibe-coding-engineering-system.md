# 工程思考｜Vibe Coding：不是AI写代码，是你在指挥一个工程系统
做项目时长出来的想法和教训。Vibe Coding 是我目前在折腾的方向。

---

这段时间一直在实践 Vibe Coding，从环境搭起，到跑通第一个 App，再到把测试、CI/CD、Subagent 协作全部串起来。越做越清醒一个事实：

**Vibe Coding 从来不是“让AI帮你偷懒”，而是一套完整的、小型的、现代工程体系。**

你以为你在写代码，其实你在搭建一个**能自我检查、能持续迭代、能稳定交付**的小系统。

## 一、我们真的在教AI写代码吗？不是。
大多数人对 AI 编程的理解停留在：
我说需求 → AI 出代码 → 复制粘贴 → 跑起来。

这不是 Vibe Coding。
这叫**一次性代码玩具**。

真正的 Vibe 是：
你定义规则 → AI 按规则执行 → 系统自动校验 → 不达标不让提交 → 自动保证质量。

它的核心不是“快”，
是**可控**。

## 二、工程上最值钱的，从来不是功能，是“不崩溃”
做项目久了都会懂：
能写出功能不厉害，
能保证**改一百次都不崩**，才厉害。

Vibe Coding 这套训练营最有价值的，不是教你做任务管理、加番茄钟，而是把四件事焊死在流程里：

1. 日志：发生什么，留痕可查
2. 测试：改之前先考一遍
3. 回归：新加功能不拆家
4. 门禁：烂代码不让提交

这四件事，就是**小型工程的全部底线**。

AI 可以帮你写 90% 的代码，
但守住底线的那个人，只能是你。

## 三、AI 时代的开发，本质是“指挥系统”
我越来越有一个体感：

**传统开发：人 → 代码。**
**AI 开发：人 → 系统 → AI → 代码。**

你不再直接跟代码搏斗。
你在跟一整套系统对话：
提示词、上下文、MCP、Skills、Subagent、测试、门禁、部署。

你的工作变成：
- 设计流程
- 制定规则
- 分配角色
- 守住质量
- 接受交付

这不叫“写代码”。
这叫**指挥一支AI开发小队**。

## 四、为什么小项目也要讲工程？因为它会长大
很多人觉得：就一个个人项目，搞什么测试、CI/CD、重构，太折腾。

但工程能力的本质是：
**在它还小的时候，就用对的结构，让它长大不会变形。**

Vibe Coding 教的就是这个：
从第一行代码开始，就按工程标准来。
- 目录结构清晰
- 数据持久化规范
- 接口干净
- 测试覆盖
- 日志齐全
- 提交有门禁

等你真的要扩展、加人、上线、维护，别人的项目已经烂成泥潭，你的依然能轻松迭代。

## 五、写在后面：什么是真正的“AI 能力”
这段实践下来，我越来越明确：

**未来不缺会写代码的人，缺敢让AI写、又能Hold住质量的人。**

AI 负责效率。
你负责：对、稳、准、安全、可维护、可迭代。

这就是下一代开发者的核心壁垒。

Vibe Coding 对我来说，不是一套课程。
它是一种**新的工作方式**：
慢一点思考，稳一点落地，用工程压住速度，用AI放大能力。

不追求完美，先占坑，后面慢慢更新。
这本身就是设计哲学。
